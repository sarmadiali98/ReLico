# Smart Home Hardware Validation Example

This directory contains the smart-home case study used for the hardware-validation part of the RELICO artifact.

It demonstrates an end-to-end workflow:

1. model the system in Timed Rebeca,
2. translate it to Lingua Franca with RELICO,
3. adapt the generated LF program to consume live sensor data, and
4. validate the verified properties during execution on an ESP32-based setup.

## What is in this directory

- `smarthome.rebeca`  
  Timed Rebeca source model for the smart-home example.

- `smarthome.property`  
  Property specification associated with the smart-home model.

- `generated_smarthome.lf`  
  The direct Lingua Franca output generated by RELICO from `smarthome.rebeca`.

- `smart_home.lf`  
  The hardware-adapted LF program used in the end-to-end experiment. This version is based on the generated LF code, but includes the extra integration needed to consume live sensor data from the serial bridge.

- `serial_bridge.py`  
  Python bridge that reads JSON sensor data from the ESP32 over serial and writes the most recent 100 samples to `sensor_data.jsonl`.

- `sample_data/sensor_data.jsonl`  
  Example sensor-data file produced by the bridge.

- `firmware/platformio.ini`  
  PlatformIO configuration for the ESP32 firmware.

- `firmware/src/main.cpp`  
  ESP32 firmware that reads the sensors and sends JSON records over serial.

- `run_scenario.sh`  
  Helper script for hardware-free replay. It writes controlled sensor values to `sensor_data.jsonl`, runs the hardware-adapted LF executable, and saves event/property logs.

- `logs/hardware-scenarios/`  
  Recorded logs for the five replayed scenarios used in the artifact. These logs contain the expected `[EVENT]`, `[INFO]`, and `[PROPERTY]` markers.

## Runtime-generated files

The following files/directories may be generated locally during execution and do not need to be committed unless explicitly included as logs:

- `sensor_data.jsonl`  
  Runtime sensor-data file read by `smart_home.lf`. During live execution, this file is continuously refreshed by `serial_bridge.py`. During replay, it is overwritten by `run_scenario.sh`.

- `bin/`, `build/`, `lib/`, `src-gen/`  
  Generated by the Lingua Franca compiler when compiling `smart_home.lf`.

- `firmware/.pio/`  
  Generated by PlatformIO when building the ESP32 firmware.

## Purpose of this example

The purpose of this example is to show that properties verified at the Timed Rebeca level can still be observed and validated in a concrete hardware-backed execution pipeline.

This example corresponds to the smart-home evaluation described in the paper, where:

- temperature below 20°C turns the heater on,
- temperature at or above 25°C turns the heater off,
- temperature above 35°C triggers a fire alert,
- motion triggers intrusion handling,
- light levels affect lighting behavior unless stealth override is active,
- fire handling has priority over intrusion handling.

## Hardware setup

This example uses an ESP32-based sensor setup with:

- ESP32 development board (`esp32dev`)
- DHT22 temperature/humidity sensor on GPIO 27
- light sensor / LDR module on GPIO 34
- PIR motion detector on GPIO 35

The ESP32 firmware sends one JSON object per second over serial at 115200 baud.

Example JSON output:

```json
{"temp":22.4,"humidity":41.0,"light":1,"motion":0}
```

## Sensor encoding

The replay workflow uses the same sensor-field names as the live ESP32 firmware:

- `temp`: temperature value in Celsius
- `humidity`: humidity percentage
- `light`: light/darkness indicator used by `smart_home.lf`
- `motion`: motion indicator from the PIR sensor

In the replay scenarios below, the values are chosen to reproduce the five paper scenarios. In particular, the replay commands use:

- `light = 0` for bright light
- `light = 1` for darkness
- `motion = 0` for no motion
- `motion = 1` for detected motion

## Software architecture

The hardware-validation pipeline has three layers:

1. **Sensor layer**  
   The ESP32 firmware samples the sensors and writes JSON messages to the serial port.

2. **Bridge layer**  
   `serial_bridge.py` reads the serial stream and continuously rewrites `sensor_data.jsonl` with the latest 100 readings.

3. **Execution layer**  
   `smart_home.lf` reads the latest sensor values from `sensor_data.jsonl` and executes the control logic, including runtime property checks.

## PlatformIO-based firmware workflow

The ESP32 firmware in this example is managed with PlatformIO and uses the Arduino framework.

### Requirements

- PlatformIO
- Python 3
- `pyserial` for the serial bridge

On macOS/Homebrew Python, use a virtual environment to avoid modifying the system Python installation:

```bash
python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install platformio pyserial
```

If PlatformIO is installed in the virtual environment, use:

```bash
python3 -m platformio run
```

instead of:

```bash
pio run
```

### Build the firmware

From the firmware directory:

```bash
cd firmware
pio run
```

If using the local virtual environment described above, run:

```bash
python3 -m platformio run
```

### Upload the firmware to the ESP32

```bash
pio run -t upload
```

If using the local virtual environment:

```bash
python3 -m platformio run -t upload
```

If your board is connected on a non-default port, configure the upload port in PlatformIO or pass the appropriate option on the command line.

### Monitor the serial output

```bash
pio device monitor -b 115200
```

If using the local virtual environment:

```bash
python3 -m platformio device monitor -b 115200
```

Expected serial output is one JSON object per second.

## Running the serial bridge

Before running the LF program in live-hardware mode, start the Python bridge so that live sensor data is written to `sensor_data.jsonl`.

Edit the serial port in `serial_bridge.py` if needed. By default it uses:

```python
PORT = '/dev/cu.usbserial-0001'
BAUD = 115200
```

Then run:

```bash
python3 serial_bridge.py
```

The bridge will:

- connect to the ESP32,
- read JSON messages from the serial port,
- keep the most recent 100 records,
- continuously rewrite `sensor_data.jsonl`,
- print a short live summary of the latest reading.

## Relation between the two LF files

Two LF files are included on purpose:

- `generated_smarthome.lf` is the raw output of RELICO.
- `smart_home.lf` is the version used for hardware validation.

This distinction is important. The hardware experiment does **not** use the raw generated file unchanged. Instead, the hardware-validation version includes the additional integration needed to read real sensor data at runtime.

## Reproducing the hardware-backed example

Recommended order:

1. Inspect or verify `smarthome.rebeca`.
2. Generate or compare `generated_smarthome.lf`.
3. Build and upload the ESP32 firmware with PlatformIO.
4. Start `serial_bridge.py`.
5. Compile the hardware-adapted LF program (`smart_home.lf`) in your Lingua Franca setup.
6. Run the generated LF executable.
7. Observe runtime event and property markers.

A typical LF compilation command from this directory is:

```bash
lfc smart_home.lf
```

This generates an executable under:

```text
bin/smart_home
```

Run it with:

```bash
./bin/smart_home
```

If your system requires the generated reactor runtime library path explicitly, run:

```bash
DYLD_LIBRARY_PATH=./lib ./bin/smart_home
```

on macOS, or the corresponding dynamic-library-path setting for your platform.

## Hardware-free scenario replay

The live hardware path reads sensor data from the ESP32 through `serial_bridge.py`.
For artifact review, the same hardware-adapted LF executable can also be run using controlled sensor values written directly to `sensor_data.jsonl`.

This replay mode is useful because it reproduces the event and property markers without requiring reviewers to physically connect the ESP32 setup, heat the temperature sensor, or trigger the PIR sensor manually.

### Build the hardware-adapted LF program

From this directory:

```bash
lfc smart_home.lf
```

This generates an executable under:

```text
bin/smart_home
```

### Run one replay scenario

The helper script has the following form:

```bash
./run_scenario.sh <scenario-name> <temp> <humidity> <light> <motion> <seconds>
```

For example, to reproduce the fire-overriding-intrusion scenario:

```bash
./run_scenario.sh scenario-3-fire-overrides-intrusion 38.0 40.0 1 1 10
```

The script writes one controlled sensor record to `sensor_data.jsonl`, runs `bin/smart_home` for the requested number of seconds, and saves the resulting log.

### Reproduce the five paper scenarios

```bash
./run_scenario.sh scenario-1-fire-emergency 38.0 40.0 0 0 8
./run_scenario.sh scenario-2-intrusion-stealth 24.0 40.0 1 1 8
./run_scenario.sh scenario-3-fire-overrides-intrusion 38.0 40.0 1 1 10
./run_scenario.sh scenario-4-normal-baseline 27.0 40.0 0 0 6
./run_scenario.sh scenario-5-cold-temperature 18.0 40.0 0 0 6
```

The included replay logs are stored under:

```text
logs/hardware-scenarios/
```

These logs demonstrate the expected `[EVENT]`, `[INFO]`, and `[PROPERTY]` markers without requiring physical hardware reproduction.

## Expected behavior

The smart-home example is designed around five validated properties:

- `FireKeepsDoorOpen`
- `FireOverridesIntrusion`
- `HeaterOffDuringFire`
- `EmergencyImpliesAlarmOrHistory`
- `LightOffDuringIntrusionOverride`

The paper evaluates five hardware scenarios:

1. **Fire Emergency**  
   High temperature (>35°C), no motion, bright light.

2. **Intrusion with Stealth Mode**  
   Motion detected, moderate temperature, darkness.

3. **Fire Overriding Intrusion (Race Condition)**  
   Fire and intrusion conditions arise together.

4. **Normal Operation (Baseline)**  
   Moderate temperature, no motion, bright light.

5. **Cold Temperature**  
   Low temperature (<20°C), no motion, bright light.

Expected runtime output includes `[EVENT]`, `[INFO]`, and `[PROPERTY]` markers indicating both control actions and property validation.

## Notes and limitations

- This directory is an **example hardware-validation workflow**, not the core translator implementation.
- The main RELICO artifact is the Timed Rebeca → Lingua Franca translator. This smart-home example shows how translated code can be used in an end-to-end deployment path.
- The hardware-backed LF program (`smart_home.lf`) includes manual integration for live sensor input and should not be confused with the direct translator output.
- The replay workflow uses controlled values written to `sensor_data.jsonl`; it is intended to reproduce the scenario-level event/property behavior, not to replace the live ESP32 validation path.
- The serial port in `serial_bridge.py` may need to be changed for your machine.
- Live hardware reproduction is optional for artifact review. The replay script and included logs provide a hardware-free way to inspect the behavior reported in the paper.
- The exact steps for building/running the LF program depend on your Lingua Franca installation.
